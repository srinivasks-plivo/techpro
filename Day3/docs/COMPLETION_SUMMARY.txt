CONCEPT #1: SQLAlchemy ORM
✓ Defined database tables as Python classes
✓ No raw SQL needed - SQLAlchemy generates it
✓ Type-safe queries using ORM methods
✓ Easy to switch databases (PostgreSQL → MySQL)
✓ Example:
  class CallLog(Base):
      call_uuid = Column(String, unique=True)
      from_number = Column(String)

CONCEPT #2: Configuration-Driven Design
✓ Menus stored in database, not hardcoded
✓ Change behavior without code changes
✓ Non-technical users can manage menus
✓ Example:
  Menu: main_menu → {digit_actions: {"1": "sales_menu"}}
  (No if/else statements needed!)

CONCEPT #3: Database Normalization
✓ Three separate tables for different purposes
✓ MenuConfiguration = Blueprint (static)
✓ CallLog = Detail records (one per call)
✓ CallerHistory = Summary profile (one per caller)

CONCEPT #4: JSON Columns
✓ Store arrays and objects in PostgreSQL
✓ menu_path = ["main_menu", "sales_menu"]
✓ digit_actions = {"1": "sales_menu", "2": "support"}
✓ user_inputs = [{"digit": "1", "timestamp": "..."}]

CONCEPT #5: Relationship Design
✓ MenuConfiguration.parent_menu_id → forms tree structure
✓ CallLog.menu_path → references menu_configurations
✓ CallerHistory.phone_number → matches CallLog.from_number

═══════════════════════════════════════════════════════════════════════════════
                       THREE MODELS EXPLAINED
═══════════════════════════════════════════════════════════════════════════════

┌─ MenuConfiguration ──────────────────────────────────────────────────────────┐
│  WHAT: The blueprint for the entire IVR system                              │
│  WHY: Configuration-driven = change menus without code changes              │
│  HOW: Store each menu as a row with digit_actions JSON mapping             │
│  WHEN: Loaded each time user needs to see a menu                           │
│  QUERIES:                                                                    │
│    - Load "main_menu" for incoming calls                                    │
│    - Get next menu from digit_actions                                       │
│    - Check if digit is valid (validation)                                   │
│  ROWS: ~20-50 rows (one per unique menu in your IVR)                        │
│  CHANGE FREQ: Rarely (only when changing menu structure)                    │
└──────────────────────────────────────────────────────────────────────────────┘

┌─ CallLog ────────────────────────────────────────────────────────────────────┐
│  WHAT: Permanent record of each individual phone call                       │
│  WHY: Audit trail, analytics, debugging                                    │
│  HOW: One row per call with all details (path taken, buttons pressed)       │
│  WHEN: Created when call ends and saved to database                         │
│  QUERIES:                                                                    │
│    - How many calls from +15551234567?                                      │
│    - What's the average call duration?                                      │
│    - Which menus are most popular?                                          │
│  ROWS: Grows with each call (millions of rows over time)                    │
│  DATA: From redis_session when call ends                                    │
└──────────────────────────────────────────────────────────────────────────────┘

┌─ CallerHistory ──────────────────────────────────────────────────────────────┐
│  WHAT: Summary profile of each unique phone number                          │
│  WHY: Quick recognition of returning callers, personalization              │
│  HOW: One row per unique phone, updated after each call                    │
│  WHEN: Checked at call start, updated at call end                          │
│  QUERIES:                                                                    │
│    - Is this a returning caller?                                            │
│    - What's their average call duration?                                    │
│    - How many times have they called?                                       │
│  ROWS: One per unique caller (grows slowly over time)                       │
│  UPDATE: total_calls += 1, total_duration += call_length                    │
└──────────────────────────────────────────────────────────────────────────────┘

                      HOW TO USE THIS FOUNDATION
═══════════════════════════════════════════════════════════════════════════════

1. QUERYING MENUS:
   from models import MenuConfiguration
   from models.database import SessionLocal
   
   db = SessionLocal()
   menu = db.query(MenuConfiguration).filter_by(
       menu_id="main_menu"
   ).first()
   print(menu.message)
   db.close()

2. SAVING A CALL:
   from models import CallLog
   from datetime import datetime
   
   call = CallLog(
       call_uuid="abc123",
       from_number="+15551234567",
       to_number="+15559876543",
       start_time=datetime.utcnow(),
       end_time=datetime.utcnow(),
       menu_path=["main_menu", "sales_menu"],
       user_inputs=[{"digit": "1"}]
   )
   db.add(call)
   db.commit()

3. UPDATING CALLER HISTORY:
   from models import CallerHistory
   
   caller = db.query(CallerHistory).filter_by(
       phone_number="+15551234567"
   ).first()
   if caller:
       caller.total_calls += 1
       caller.total_duration += 300
       db.commit()
   else:
       # Create new caller
       new_caller = CallerHistory(phone_number="+15551234567")
       db.add(new_caller)
       db.commit()

🎯 THREE TABLES, ONE PURPOSE:
   ├─ MenuConfiguration = How should the system behave?

   └─ When call ends: temporary → permanent
